Version control is a system that records changes to files over time, allowing you to recall specific versions later. Think of it as a time machine for your code.

Why Version Control Matters

Without Version Control:

  • Files get overwritten accidentally
  • No way to track who made what changes
  • Difficult to collaborate with others
  • No backup of your work

With Version Control:

  • Track every change to your files
  • Revert to any previous version
  • Collaborate safely with others
  • Complete history of your project

Types of Version Control

  1. Local Version Control - Files stored on your computer
  2. Centralized Version Control - Single server stores all files
  3. Distributed Version Control - Every user has a complete copy

Git is a distributed version control system, which means every user has a complete copy of the repository.

Positive : Git is the most popular version control system in the world, used by millions of developers and companies.

Before we can use Git, we need to install it on your computer.

Windows Installation

  1. Download Git for Windows:
    • Go to https://git-scm.com/download/win
    • Download the latest version
    • Run the installer
  2. Installation Options:
    • Use Git from Git Bash only (recommended)
    • Use Git from Windows Command Prompt
    • Use Git and optional Unix tools from Windows Command Prompt
  3. Verify Installation:
    git --version

macOS Installation

Option 1: Using Homebrew

brew install git

Option 2: Download from git-scm.com

  • Visit https://git-scm.com/download/mac
  • Download and install

Linux Installation

Ubuntu/Debian:

sudo apt-get update
sudo apt-get install git

CentOS/RHEL:

sudo yum install git

Verify Installation

After installation, verify Git is working:

git --version

You should see output like: git version 2.34.1

Positive : If you see a version number, Git is installed correctly!

Now that Git is installed, we need to configure it with your identity.

Set Your Identity

Git needs to know who you are for commit messages:

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Verify Configuration

Check your Git configuration:

git config --list

Additional Configuration (Optional)

Set default editor:

git config --global core.editor "notepad"  # Windows
git config --global core.editor "nano"      # Linux/macOS

Set default branch name:

git config --global init.defaultBranch main

Negative : Never use fake email addresses in Git configuration. Use your real email for professional projects.

Configuration Levels

  • System level (--system): Applies to all users on the computer
  • Global level (--global): Applies to all repositories for current user
  • Local level (no flag): Applies only to current repository

Let's create your first Git repository and learn the basic workflow.

Initialize a Repository

  1. Create a new directory:
    mkdir my-first-project
cd my-first-project
  2. Initialize Git repository:
    git init
  3. Check repository status:
    git status

Understanding Git Status

The git status command shows:

  • Which branch you're on
  • Whether your working directory is clean
  • Which files are staged for commit
  • Which files are untracked

Your First Commit

  1. Create a file:
    echo "# My First Project" > README.md
  2. Add file to staging area:
    git add README.md
  3. Commit the changes:
    git commit -m "Add README file"

Understanding the Workflow

  1. Working Directory - Where you edit files
  2. Staging Area - Where you prepare files for commit
  3. Repository - Where Git stores the committed files

Positive : Think of the staging area as a "preparation area" where you get files ready for their permanent snapshot.

Let's learn the essential Git commands you'll use every day.

Checking Status and History

Check repository status:

git status

View commit history:

git log

View commit history (compact):

git log --oneline

Adding and Committing Files

Add specific file:

git add filename.txt

Add all files:

git add .

Commit with message:

git commit -m "Your commit message"

Add and commit in one step:

git commit -am "Your commit message"

Viewing Changes

See what files have changed:

git status

See the actual changes:

git diff

See staged changes:

git diff --staged

Best Practices for Commits

  1. Write clear commit messages:
    • Use present tense ("Add feature" not "Added feature")
    • Be descriptive but concise
    • Start with a capital letter
  2. Make atomic commits:
    • Each commit should represent one logical change
    • Don't mix unrelated changes in one commit
  3. Commit frequently:
    • Commit when you complete a small feature
    • Commit when you fix a bug
    • Don't wait until everything is perfect

Negative : Avoid commit messages like "fix" or "update". Be specific about what you changed.

Branches allow you to work on different features or versions of your code simultaneously.

Understanding Branches

A branch is like a separate timeline for your project. You can:

  • Work on new features without affecting the main code
  • Experiment with changes safely
  • Collaborate with others on different features

Branch Commands

List all branches:

git branch

Create a new branch:

git branch feature-name

Switch to a branch:

git checkout feature-name

Create and switch to new branch:

git checkout -b feature-name

Switch to main branch:

git checkout main

Merging Branches

Merge a branch into current branch:

git merge feature-name

Delete a branch after merging:

git branch -d feature-name

Branch Workflow Example

  1. Start on main branch:
    git checkout main
  2. Create feature branch:
    git checkout -b add-login-feature
  3. Make changes and commit:
    echo "Login functionality" > login.js
git add login.js
git commit -m "Add login feature"
  4. Switch back to main:
    git checkout main
  5. Merge feature branch:
    git merge add-login-feature

Positive : Using branches is a best practice that keeps your main code stable while allowing experimentation.

GitHub is a web-based platform that hosts Git repositories and provides collaboration features.

What is GitHub?

  • Repository hosting - Store your Git repositories online
  • Collaboration tools - Work with others on projects
  • Issue tracking - Manage bugs and feature requests
  • Pull requests - Review and merge code changes
  • Project management - Organize and track project progress

Creating a GitHub Account

  1. Visit GitHub.com
  2. Click "Sign up"
  3. Enter your email and password
  4. Choose your plan (Free plan is sufficient for most users)
  5. Complete the setup process

Creating Your First Repository

  1. Click the "+" icon in the top right
  2. Select "New repository"
  3. Fill in repository details:
    • Repository name
    • Description (optional)
    • Public or private
    • Initialize with README (recommended)
  4. Click "Create repository"

Connecting Local Repository to GitHub

Add remote repository:

git remote add origin https://github.com/username/repository-name.git

Push your code to GitHub:

git push -u origin main

Verify remote connection:

git remote -v

Positive : GitHub makes it easy to share your code and collaborate with others around the world.

Learn how to sync your local repository with GitHub.

Pushing to GitHub

Push your commits to GitHub:

git push origin main

Push a new branch:

git push -u origin branch-name

Push all branches:

git push --all

Pulling from GitHub

Pull latest changes:

git pull origin main

Fetch changes without merging:

git fetch origin

Pull specific branch:

git pull origin branch-name

Understanding Push and Pull

  • Push - Upload your local changes to GitHub
  • Pull - Download changes from GitHub to your local repository
  • Fetch - Download changes but don't merge them yet

Common Workflow

  1. Start your day:
    git pull origin main
  2. Make changes and commit:
    git add .
git commit -m "Your changes"
  3. Push to GitHub:
    git push origin main

Handling Conflicts

Sometimes when you pull, Git can't automatically merge changes:

  1. Git will tell you there's a conflict
  2. Open the conflicted files
  3. Look for conflict markers (<<<<<<<, =======, >>>>>>>)
  4. Edit the files to resolve conflicts
  5. Add and commit the resolved files

Negative : Always pull before pushing to avoid conflicts with other people's changes.

Learn how to work effectively with others using GitHub's collaboration features.

Forking a Repository

What is forking?

  • Creating your own copy of someone else's repository
  • Allows you to make changes without affecting the original
  • Common way to contribute to open source projects

How to fork:

  1. Go to the repository on GitHub
  2. Click the "Fork" button
  3. Choose your account
  4. GitHub creates your copy

Cloning a Repository

Clone your forked repository:

git clone https://github.com/your-username/repository-name.git

Clone any public repository:

git clone https://github.com/username/repository-name.git

Working with Remote Repositories

Add the original repository as upstream:

git remote add upstream https://github.com/original-owner/repository-name.git

Keep your fork updated:

git fetch upstream
git merge upstream/main
git push origin main

Pull Requests

What are pull requests?

  • A way to propose changes to a repository
  • Allows code review before merging
  • Standard way to contribute to projects

Creating a pull request:

  1. Make changes in your fork
  2. Push changes to your fork
  3. Go to your fork on GitHub
  4. Click "New Pull Request"
  5. Fill in the description
  6. Submit the pull request

Code Review Process

Reviewing pull requests:

  1. Read the description
  2. Look at the changes
  3. Test the changes if possible
  4. Leave comments and suggestions
  5. Approve or request changes

Responding to review comments:

  1. Make the requested changes
  2. Push the changes to your branch
  3. The pull request updates automatically

Positive : Pull requests are a great way to learn from others and improve your code through feedback.

Follow these best practices to use Git and GitHub effectively.

Commit Message Guidelines

Good commit messages:

  • "Add user authentication feature"
  • "Fix login form validation bug"
  • "Update README with installation instructions"

Bad commit messages:

  • "fix"
  • "update"
  • "stuff"

Repository Organization

File structure:

project-name/
├── README.md
├── .gitignore
├── src/
├── docs/
└── tests/

README.md should include:

  • Project description
  • Installation instructions
  • Usage examples
  • Contributing guidelines

.gitignore File

Create a .gitignore file to exclude unnecessary files:

# Dependencies
node_modules/
vendor/

# Build files
dist/
build/

# IDE files
.vscode/
.idea/

# OS files
.DS_Store
Thumbs.db

Branch Naming Conventions

Good branch names:

  • feature/user-authentication
  • bugfix/login-validation
  • hotfix/security-patch

Bad branch names:

  • new-feature
  • fix
  • branch1

Security Best Practices

  1. Never commit sensitive information:
    • Passwords
    • API keys
    • Database credentials
  2. Use environment variables for secrets
  3. Regularly update dependencies
  4. Use strong authentication

Negative : If you accidentally commit sensitive information, immediately change the credentials and consider the old ones compromised.

Learn how to solve common Git and GitHub problems.

Common Git Issues

Undo last commit (keep changes):

git reset --soft HEAD~1

Undo last commit (discard changes):

git reset --hard HEAD~1

Change last commit message:

git commit --amend

Recover deleted file:

git checkout HEAD -- filename

Common GitHub Issues

Authentication problems:

  • Check your username and email
  • Verify your SSH keys or personal access tokens
  • Ensure you have the correct permissions

Push rejected:

git pull origin main
git push origin main

Large file issues:

  • Use Git LFS for large files
  • Remove large files from history if needed
